# Connector contracts

## Capability-based design

A connector is a configured adapter that may implement one or more independent
capabilities:

```ts
interface KnowledgeSource {
  discover(request: DiscoveryRequest): AsyncIterable<DiscoveredItem>;
  load(reference: ExternalReference): Promise<SourceDocument>;
}

interface CaseSource {
  discoverCases(request: CaseDiscoveryRequest): AsyncIterable<CaseReference>;
  loadCase(reference: CaseReference): Promise<CaseSnapshot>;
}

interface AttachmentSource {
  openAttachment(reference: AttachmentReference): Promise<AttachmentStream>;
}

interface AnalysisDestination {
  findPublication(marker: PublicationMarker): Promise<ExistingPublication | null>;
  publish(request: PublishRequest): Promise<PublicationReceipt>;
}

interface WebhookAdapter {
  verify(request: RawWebhookRequest): Promise<VerifiedWebhook>;
  translate(event: VerifiedWebhook): Promise<DomainEvent[]>;
}
```

The final signatures will use shared cancellation, pagination, rate-limit, and typed-error
primitives from `connector-sdk`.

`DiscoveredItem` carries a connector-owned external revision/fingerprint when available,
such as a Git blob OID, ETag, API version, update sequence, or stable content checksum.
The fingerprint is opaque to the core and is used as a cheap first change check. A source
that cannot provide one may require loading content and comparing the normalized hash.

## Neutral case contracts

The connector SDK owns explicit schemas for case lifecycle, actors, ordered messages,
public/internal/system visibility, HTML and plain-text bodies, timestamps, attachment
references, access metadata, and optional resolution semantics. These schemas implement
the normative shape in `02-domain-and-workflows.md` and are finalized before a Jitbit
adapter is written.

## Portability rules

- Core identifiers are generated by CaseWeaver. External IDs are scoped by connector.
- Connector metadata is JSON validated by connector-owned schemas.
- Core code cannot branch on connector type.
- Authentication is represented by secret references resolved at runtime.
- Cursors are opaque to the core but versioned by the connector.
- External revision/fingerprint values are opaque, persisted, and compared within the
  configured connector instance.
- Connectors declare capabilities and supported webhook event types.
- Rate-limit information and retry-after values are surfaced, not swallowed.
- Deletion and permission-loss events are distinguishable.
- Discovery must be cheap and must not invoke an AI provider.

## Knowledge-source configuration

Every configured knowledge-source instance independently defines:

- connector instance and source capability,
- enabled state,
- target knowledge collection,
- normalization and chunking profiles,
- synchronization mode: manual, cron, interval polling, webhook, or a supported
  combination,
- cron timezone, jitter, overlap policy, and maximum duration,
- incremental cursor and optional periodic full-rescan policy,
- deletion behavior and retention,
- and source-specific filters.

Scheduling belongs to the configured source instance, not to the connector package.
Connectors expose capabilities; the scheduler decides when to invoke them.

## Destination configuration

Publication profiles reference configured `AnalysisDestination` instances. A destination
is not assumed to be the same connector that supplied the case. Profiles may use no
destination, one destination, or policy-controlled multiple destinations without
changing the analysis engine.

## Reference adapters

### Git/Markdown

Implements `KnowledgeSource`. Supports local repositories and authenticated Git URLs,
branch or tag selection, path filters, and stable source URLs. Docusaurus is initially a
Markdown convention layered on this adapter, not a separate core dependency.

### Jitbit

May implement `KnowledgeSource`, `CaseSource`, `AttachmentSource`,
`AnalysisDestination`, and `WebhookAdapter`. It validates the contracts but does not
define them.

### Future Odoo

An Odoo connector should map its helpdesk records to the same Case and message models.
Adding it must require a connector package, configuration schema, and contract tests
only. No database migration or analysis-engine modification should be necessary unless
Odoo exposes a genuinely new domain capability.

## Connector development kit

The SDK must provide:

- configuration-schema helpers,
- mock secret resolution,
- pagination and retry test utilities,
- contract test suites,
- fixture builders,
- normalized HTML-to-text utilities,
- and conformance documentation.

Contract tests cover cursor continuation, duplicate discovery, updates, deletions,
attachments, rate limits, typed failures, and publication idempotency.

The conformance fixture set must include at least:

- a Jitbit-shaped case,
- an Odoo-shaped case with different lifecycle and actor fields,
- missing optional capabilities,
- internal and public messages,
- and an external system that cannot provide a monotonic revision token.
